# VoxFlow TN 模型方案(修订版):Qwen3-0.6B Span-Edit TN

> 版本:v1.1(2026-07-04)。基于原始方案评审修订,适配当前开发机(8× H200)第一期落地:
> **第一期数据不走 WFST 反向合成,改为"本地大模型造语境 + 程序化 verbalizer 造 gold 读法"**,
> 微调与离线验证全部在本机完成;延迟压测与线上集成(L20 / TurboMind / VoxFlow)留待二期。

目标:基于 Qwen3-0.6B 的编辑式生成 TN 模型,端到端延迟 P99 < 50ms(L20, bf16 起步),数字类零幻觉,可灰度替换现有 WFST 主链路。

总体架构(线上推理链路,最终态):

```
原文 → Stage A 前置规则(Aho-Corasick/WFST 高置信命中 → PUA 占位)
     → Qwen3-0.6B Span-Edit 生成(greedy)
     → 输出解析 + 校验(合法读法验证)
     → 拼接还原(含 PUA 占位符回填 / .tnc 租户 overlay)
     → 失败/超时 fallback:纯 WFST 全链路
```

## 0. 评审结论与第一期范围界定

### 0.1 评审结论(相对原方案的关键修订)

1. **数据源改为 LLM 造数据后,最大风险从"覆盖不足"转移为"gold 读法标签噪声"。**
   数字读法错误恰恰是 P0 红线,绝不能让 LLM 自由生成读法。修订原则:
   **LLM 只负责语境,gold 读法一律程序化生成**——写一个轻量 Python verbalizer 库
   (数字/日期/时间/金额/电话/百分比等,可基于 cn2an 等做 building block,几百行代码),
   承担原方案中 verbalizer FST 的角色。LLM 的职责收窄为:
   (a) 生成带槽位的多样化真实句子;(b) 歧义难例的"选择题式" judge;(c) 评测时的辅助裁判。
   这保留了 WFST 反向合成"gold 天然正确"的核心性质,只是把 FST 换成 Python 函数,成本低一个量级。

2. **锚点唯一化规则两侧解耦,降低脆性。**
   原方案要求"数据合成与解析器按同一规则扩窗",这是一个隐性强耦合。修订为:
   - 解析器契约放宽:从上一条编辑结束位置向后,取**第一个**匹配位置即可,不要求剩余文本内全局唯一;
   - 数据侧仍按"最短唯一化(NSW 本体,不唯一则 ±1 字扩窗)"生成锚点;
   - 这样模型锚点略有出入时(多扩/少扩一个字)解析仍成立,失败面只剩"锚点完全找不到",全部由 ParseError → fallback 兜底。

3. **分隔符碰撞风险**:`->` 理论上可出现在原文(代码、箭头符号)。一期不改 spec
   (碰撞率极低且 ParseError 兜底),但解析器必须以"行内**最后一个** ` -> `"作为分隔(锚点里含 `->` 时仍可解析),
   并把该类失败计入监控。若实测碰撞率不可忽略,二期换 `\t` 分隔。

4. **训练框架**:本机 transformers 5.8.1(很新),ms-swift / LLaMA-Factory 大概率有版本兼容问题。
   0.6B 单任务 SFT 用**自研 HF Trainer 脚本**(~200 行:loss 只算 completion、可选 packing)反而最稳、可控性最好。
   若后续要上 DPO 再评估 TRL。

5. **本机(H200)延迟数据不代表 L20**。本机只验证:正确性指标、输出 token 数分布、decode 步数;
   P99 延迟结论必须在 L20 + TurboMind 上重测。二期事项。

6. **规模节奏**:先 5 万条把"造数 → 校验 → 训练 → 评测"全链路跑通并做第一轮错误分析,再放大到 50 万。
   LLM 造数的同质化风险(模板化、实体单一)必须用条件化生成 + 去重压制(见 2.4)。

7. **评测替代方案**:5k 人工标注集一期没有。替代:独立种子/独立 prompt 的合成测试集(与训练数据 pipeline 隔离)
   + 大模型 judge + **人工抽检 500 句**。明确:该评测只用于迭代选型,**不能作为上线依据**;上线前必须补人工集。

8. **待验证假设(W1 落实)**:Qwen3 词表中确有可用的保留/unused special token 充当 `<|tn|>` `<|sep|>`
   且各为单 token;Qwen3-0.6B-Base 权重可下载。以 tokenizer 实测为准,不行则退而用普通短字符串前缀(代价:多几个 prefill token)。

### 0.2 第一期(本机)范围

| 在范围内 | 不在范围内(二期) |
|---|---|
| 输出格式 spec 冻结 + parse_and_apply + fuzz | Stage A 前置规则 / PUA 占位链路 |
| Python verbalizer 库(gold 读法生成与反向校验) | WFST 反向合成 pipeline、NeMo 语法复用 |
| DeepSeek API(deepseek-v4-flash)造数据 5 万 → 50 万 | 5k 人工标注测试集(启动标注可并行) |
| Qwen3-0.6B-Base 全参 SFT(Stage-1 + Stage-2 难例强化) | TurboMind/L20 部署、量化、投机解码 |
| 离线评测框架:句准率/Span P·R/数字错读率/解析失败率/分桶 | 影子模式、灰度、VoxFlow 集成 |
| H200 上 vLLM 起服务做正确性冒烟 + token 数统计 | P99 延迟压测(必须 L20) |

## 1. 输出格式 Spec(先定死,数据和训练都依赖它)

### 1.1 anchor-text 编辑格式

放弃字符下标(LLM 数不准,错一位整句报废),用原文锚点:

```
输入: 会议定在18号下午3:30,预算¥1200万。
输出: 18号->十八号
      3:30->三点三十分
      ¥1200万->一千二百万元
      <eos>

输入: 今天天气不错。
输出: <eos>
```

规则:

- 每行一条编辑:`原文片段->替换读法`,按**原文出现顺序**输出。
- 定位靠字符串匹配:解析器从上一条编辑的结束位置向后搜索锚点,取第一个匹配(见 0.1-2),
  天然解决同一 NSW 多次出现且读法不同的问题(如"他18岁,住18号楼")。
- 数据侧锚点最短唯一化:默认取 NSW 本体;若在剩余文本中不唯一,向左右各扩一个字。
- 无改写句子输出仅为 `<eos>`(1 个 decode step,P50 延迟关键)。Base 模型用 `<|endoftext|>` 充当 eos,训练/推理 stop 配置一致。
- PUA 占位符区间对模型不可见(二期 Stage A);模型输出中禁止出现 PUA 码位,解析器遇到即判失败。

### 1.2 解析器契约(先写好并冻结)

独立模块 `tn/parser.py`:`parse_and_apply(src, model_output) -> normalized_text | ParseError`。

- 行分隔按 `\n`;行内以**最后一个** `->` 分隔锚点与读法。
- 失败类型全部枚举并打点:锚点未找到 / 顺序倒置 / 读法含非法字符(仅允许汉字 + 少量标点)/ 含 PUA / 空编辑。
- 配 fuzz 测试:随机原文 + 随机合法编辑脚本 round-trip 必须无损;随机损坏的输出必须稳定返回 ParseError 而非异常。
- **解析失败率是核心监控指标**,目标 < 0.1%。

### 1.3 Prompt 设计

Base 模型 + 极简固定前缀,不用 chat template:

```
<|tn|>{原文}<|sep|>
```

`<|tn|>` `<|sep|>` 用 Qwen3 词表保留 token 充当(W1 用 tokenizer 实测确认具体 token,要求单 token、训练语料中不出现)。不扩词表、不动 embedding。固定前缀开 prefix caching 后 prefill 只算原文部分。

## 2. 数据准备(第一期:LLM 造语境 + 程序化 gold)

### 2.1 核心 pipeline:三段式造数

```
① 采样规格:semiotic class × 域(新闻/对话/客服/UGC/小说)× 句长档 × 读法变体
② 程序化渲染:Python verbalizer 生成 (书面形式, gold 读法) 对
   例:NUMBER(1200万, 语境=金额) → ("¥1200万", "一千二百万元")
③ LLM 造句:给定书面形式 + 域/风格约束,生成包含该片段的自然句子
   (要求原样保留书面形式,生成后用字符串匹配强校验,不匹配即弃)
④ 按 1.1 规则自动生成锚点与目标编辑脚本
```

关键点:**gold 读法从不经过 LLM**。LLM 只产出语境句子,书面形式片段在句中的存在性用精确匹配校验,读法由 ② 的确定性代码给出。

### 2.2 semiotic class 覆盖(verbalizer 库的实现清单)

NUMBER / DATE / TIME / MONEY / PHONE / MEASURE / PERCENT / FRACTION / SCORE / VERSION / RANGE / SERIAL。

- 同一书面形式多合法读法(如 "18" → {十八, 一八, 幺八})按语境类型条件采样(电话语境采"幺八",数量语境采"十八");语境类型即 ① 的采样标签,LLM 造句时按该语境生成。
- 每类维护:书面形式模板(含符号变体:`¥/元/块`、`:/ 点`、全半角)、gold 渲染函数、**反向校验函数**(给定书面形式+读法判断是否合法,入库与在线校验共用)。
- 难类(TIME、混合单位、SCORE、RANGE)采样权重 ×2–3。

### 2.3 蒸馏难例(~15%)

歧义样本用 DeepSeek API(`deepseek-v4-flash`;造句用非思考模式,难例仲裁可开思考)做**选择题式** judge:给定句子 + verbalizer 生成的候选读法集合,让大模型选,不自由生成。重点类目:

- 数字串:电话/编号 vs 数量("110" 报警 vs 一百一十)
- 年份 vs 数量("2024年" vs "2024个")
- 型号/代号不展开("iPhone 15"、"P100"、"Boeing 737")——同时是重要负样本
- 英文缩写与单位("3km/h"、"GDP"、"5G")

judge 用 self-consistency(n=3 投票),不一致样本进人工抽检池;人工抽检 5%。

### 2.4 负样本(~25%)与多样性控制

- 负样本一半普通句、一半**易误触句**(含数字但不需改写:型号、代码、英文混排、类 NSW 专名)。目标输出仅 `<eos>`。过度触发比漏触发更伤听感。
- 普通句可直接取真实语料(如 WenetSpeech 转写、公开新闻语料)原句,经"确认无 NSW"过滤(用 verbalizer 的书面形式检测器反查)。
- **同质化压制**:造句 prompt 条件化(域 × 风格 × 句长 × 实体类型),每批换种子实体表;入库前按字符 5-gram Jaccard 去重(阈值 0.7);监控实体/模板重复度。

### 2.5 配比与规模

| 来源 | 占比 |
|---|---|
| 三段式合成(② 程序化 gold) | ~60% |
| 大模型选择题式蒸馏难例 | ~15% |
| 负样本(无改写) | ~25% |

节奏:**先 5 万条全链路跑通 + 训一版 + 错误分析,再放大到 50 万**。句长分布对齐线上真实 TTS 请求(重点 5–60 字,含标点切块短句)。

### 2.6 schema 与入库三道校验

```jsonl
{"src": "会议定在18号下午3:30,预算¥1200万。",
 "edits": [["18号","十八号"],["3:30","三点三十分"],["¥1200万","一千二百万元"]],
 "meta": {"classes":["DATE","TIME","MONEY"], "source":"synth", "domain":"news", "ctx":"amount"}}
```

(a) `parse_and_apply` round-trip 成功;(b) 读法全部落在合法字符集(汉字+少量标点);(c) gold 读法通过 verbalizer 反向校验函数。

### 2.7 测试集(与训练 pipeline 隔离)

- **合成盲测集 3–5k**:独立随机种子、独立造句 prompt 措辞、独立实体表,按 class × 域分层。
- **人工抽检集 500 句**:从盲测集与真实语料中抽,人工核对 gold,作为"评测的评测"。
- **难例回归集**:judge 不一致样本、fuzz 边界样本、训练迭代中的 badcase,滚动累积。
- 冻结版本管理,评测报告绑定测试集版本号。
- 二期启动 5k 线上真实请求人工标注(双标 + 仲裁),作为上线唯一 ground truth。

## 3. 训练

### 3.1 基本配置(本机 8× H200)

0.6B 全参 SFT,不用 LoRA:

- 框架:**自研 HF Trainer 脚本**(transformers 5.8.1 已装;ms-swift/LLaMA-Factory 对 5.x 兼容性存疑,不押注)
- 底座 `Qwen3-0.6B-Base`(W1 下载)
- 硬件:单卡 H200 即可(0.6B × 50 万条 × 3 epoch 约 1–2 小时);多卡留给造数服务和并行实验
- 超参起点:lr 2e-5,cosine,warmup 3%,global batch 256,max_len 512,bf16,flash-attn
- **Loss 只算 completion**(`<|sep|>` 之后),prompt mask
- packing 一期可不做(数据量小、卡快);吞吐成瓶颈再加

### 3.2 两阶段配方

1. **Stage-1 SFT**:全量混合数据 2–3 epoch。
2. **Stage-2 难例强化**:Stage-1 模型在训练分布外合成数据上采样(temperature 0.8, n=4),收集出错样本,与等量正确难例混合,再 fine-tune 0.5–1 epoch(lr 5e-6)。廉价版 rejection-sampling;句准率进 99%+ 平台期后若剩余错误是偏好型歧义再考虑 DPO。

### 3.3 训练期评测钩子

每个 checkpoint 自动跑盲测集 + 解析失败率 + 按类分桶,结果落盘固定表格。选型优先级:**数字类错读率(P0)→ 句准率 → 过度触发率**,不看 loss。

## 4. 评测体系

### 4.1 指标(在合成盲测集上,一期)

| 指标 | 定义 | 一期目标 |
|---|---|---|
| 句准率 | 整句结果与 gold 完全一致 | ≥ 98.5% |
| Span P / R | 编辑级 precision(过度触发)/ recall(漏触发) | P ≥ 99%,R ≥ 98% |
| 数字错读率 | NUMBER/MONEY/PHONE 读法错误 | ≤ 0.05%(P0 红线) |
| 解析失败率 | ParseError 占比 | ≤ 0.1% |
| 分桶准确率 | class × 域 | 无单桶 < 95% |

注:一期指标基于合成盲测集 + 500 句人工抽检,只用于迭代;上线判定必须等 5k 人工集(二期)。
对拍基线:现有 WFST 链路 + 一家商业 TTS 前端 API(二期接入)。

### 4.2 线上评测(二期)

影子模式旁路 diff + 每天 200 条人工审,1–2 周;监控四件套:解析失败率、fallback 率、P50/P99、diff 率。

## 5. 推理与延迟优化

### 5.0 一期(本机 H200,只验证正确性与 token 经济性)

- vLLM 起 0.6B 服务,greedy,统计:输出 token 数分布、无编辑句子是否稳定 1-step `<eos>`、解析失败率。
- 记录 decode 步数分布,作为 L20 延迟的推算输入(步数 × L20 单步时延)。

### 5.1 二期分层(L20 上线路径,原方案保留)

**第一层(上线必须,预期 P99 ≈ 35–45ms):** LMDeploy TurboMind bf16 + CUDA Graph;greedy;prefix caching;`<eos>` early-stop;按标点/韵律短语切 20–40 字块,跨块数字 2–3 字 lookahead。

**第二层(P99 ≈ 25–35ms):** W8A8/FP8 量化(量化后重跑全量评测,盯数字类回退);Rust fast tokenizer + 预热。

**第三层(可选):** EAGLE3 自投机(改写内容可预测性高,draft 接受率好);token 级约束解码(见 5.2)。

### 5.2 正确性保障:先"生成后校验",再"约束解码"

1. **v1 生成后校验**:每条编辑用 verbalizer 反向校验函数验证「书面形式 → 读法」合法;非法则该 span 回退默认读法(而非整句 fallback)。一期即实现(评测框架同款代码)。
2. **v2 token 级约束**:若换 vLLM 线上链路,用 xgrammar 挂 per-span trie mask;额外收益是允许更激进的量化与投机。

### 5.3 服务集成(VoxFlow,二期)

Python Worker 内嵌 TurboMind,Go Scheduler ZMQ 派发,TN 挂 L0;batch=1 保延迟,continuous batching max_batch=4;35ms 超时熔断走 WFST fallback;单 L20 数百 QPS,一卡 + 热备覆盖前端流量。

## 6. 里程碑(第一期,本机,4 周)

| 周 | 交付物 |
|---|---|
| W1 | spec 冻结;`parser.py` + fuzz;verbalizer 库(NUMBER/DATE/TIME/MONEY/PHONE 五大类);vLLM 造数服务起起来;首批 5k 样本冒烟 |
| W2 | verbalizer 补全 12 类;造数到 5 万(含负样本/难例 judge);盲测集 v0;训练脚本跑通,首版模型 + 首份分桶报告 |
| W3 | 错误分析 → 数据 v2(补负样本/难例,放大到 20–50 万);Stage-1 v2 + Stage-2 难例强化;人工抽检 500 句 |
| W4 | 迭代至指标达标;生成后校验接入评测;vLLM 正确性冒烟 + token 统计;一期总结报告,冻结 checkpoint,规划二期(L20/TurboMind/影子) |

### 风险与预案

- **LLM 造句同质化 / 语境失真** → prompt 条件化 + 换种子实体 + n-gram 去重;抽检发现失真类目改用真实语料挖槽插入
- **锚点匹配失败率高** → 数据侧扩窗到 ±2 字;解析器保持宽松第一匹配
- **过度触发压不下来** → 负样本占比提到 35%,难例负样本定向合成
- **合成盲测集与真实分布偏差**(一期固有风险)→ 500 句人工抽检兜底;尽早启动二期 5k 人工集
- **0.6B 容量不足** → 一期只做 TN,G2P 沿用现有 BERT 前端;备选升 1.7B-Base(本机已缓存)对比一版
- **transformers 5.x 踩坑** → 训练脚本自研、依赖面小;必要时 pin 到 4.x 另建 venv
